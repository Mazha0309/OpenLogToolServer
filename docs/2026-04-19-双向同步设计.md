# 全量双向同步设计

## 概述

为 OpenLogTool 客户端与 OpenLogToolServer 增加真正的双向同步能力，覆盖以下数据：

- 日志（logs）
- 词典（device / antenna / qth / callsign）
- 呼号-QTH 历史（callsign_qth_history）
- 历史归档（history）

同步要求如下：

- 客户端和服务端都支持上述数据的同步读写
- 冲突按“**更新时间较新者优先**”处理
- 删除采用“**软删除同步**”，避免误删被直接扩散为不可恢复的硬删除
- 实时同步、间隔同步、手动同步都走同一套双向同步协议
- 在建立连接后自动同步一次（适用于 realtime 和 interval）

## 当前问题

### 客户端

当前 Flutter 客户端的同步实现存在这些结构性问题：

1. `SyncProvider.triggerSync()` 之前是空实现，自动触发链路无法真正同步
2. `pushSync()` / `pullSync()` 仍然是分离式设计，且 `pullSync()` 没有被接入实际流程
3. `settings_panel.dart` 中 `_performSync(...)` 长期承担了真实同步逻辑，导致自动同步依赖 UI 路径
4. 本地 SQLite 数据格式不统一：
   - `LogEntry` 没有统一的同步主键、更新时间、删除标记
   - `DictionaryItem` 只有局部字段，没有统一同步元数据
   - `history` 目前更像本地快照，没有服务端对应结构
   - `callsign_qth_history` 仅有追加记录，没有冲突语义

### 服务端

当前服务端虽然已有 `/logs/sync/bidirectional` 入口，但仍然只是半成品：

1. 仅对日志做了部分双向合并
2. 词典与 QTH 历史仍是“客户端 bulk upsert，服务端整包返回”式处理
3. `history` 归档尚未纳入服务端同步模型
4. 当前仓储层缺少统一的：
   - `findSince(...)`
   - `upsert(...)`
   - `softDelete(...)`
   - 冲突比较逻辑

## 设计目标

本次同步设计不追求“抽象上最漂亮”，而是追求这几个结果：

1. 同一套协议覆盖四类数据
2. 客户端和服务端都使用统一同步字段
3. 自动同步、手动同步、定时同步共享同一个底层实现
4. 删除不会丢失数据恢复能力
5. 数据格式先规范，再做真正双向同步

## 统一同步模型

### 通用字段

四类同步实体统一补充以下字段：

| 字段 | 说明 |
|---|---|
| `id` | 全局唯一主键，使用 UUID，客户端生成，服务端保留 |
| `createdAt` | 创建时间，ISO 8601 |
| `updatedAt` | 最后更新时间，ISO 8601 |
| `deletedAt` | 软删除时间，`null` 表示未删除 |
| `userId` | 数据归属用户 |
| `sourceDeviceId` | 可选，记录最初来源设备 |

### 各实体格式

#### 1. 日志

```json
{
  "id": "uuid",
  "userId": "uuid",
  "sourceDeviceId": "device-id",
  "time": "09:32",
  "controller": "BG5XXX",
  "callsign": "BG5YYY",
  "report": "59",
  "qth": "Shanghai",
  "device": "FT-891",
  "power": "50W",
  "antenna": "GP",
  "height": "10m",
  "createdAt": "2026-04-19T09:32:00.000Z",
  "updatedAt": "2026-04-19T09:32:00.000Z",
  "deletedAt": null
}
```

#### 2. 词典

```json
{
  "id": "uuid",
  "userId": "uuid",
  "type": "device",
  "raw": "FT-891",
  "pinyin": "",
  "abbreviation": "",
  "createdAt": "2026-04-19T09:32:00.000Z",
  "updatedAt": "2026-04-19T09:32:00.000Z",
  "deletedAt": null
}
```

#### 3. 呼号-QTH 历史

```json
{
  "id": "uuid",
  "userId": "uuid",
  "callsign": "BG5YYY",
  "qth": "Shanghai",
  "recordedAt": "2026-04-19T09:32:00.000Z",
  "createdAt": "2026-04-19T09:32:00.000Z",
  "updatedAt": "2026-04-19T09:32:00.000Z",
  "deletedAt": null

```

#### 4. 历史归档

服务端也要实现 `history` 表。`history` 不再只是客户端本地快照，而是可同步资源：

```json
{
  "id": "uuid",
  "userId": "uuid",
  "name": "20260419 09:32 (35条记录)",
  "logsData": "json-string-or-normalized-array",
  "logCount": 35,
  "createdAt": "2026-04-19T09:32:00.000Z",
  "updatedAt": "2026-04-19T09:32:00.000Z",
  "deletedAt": null
}
```

## 客户端格式规范

为了支持双向同步，客户端本地 SQLite 需要规范化：

### 1. logs 表

新增字段：

- `id TEXT UNIQUE`
- `created_at TEXT NOT NULL`
- `updated_at TEXT NOT NULL`
- `deleted_at TEXT NULL`
- `source_device_id TEXT NULL`

原本的自增 `INTEGER PRIMARY KEY AUTOINCREMENT` 可以保留作本地索引，但不能再作为同步主键。

### 2. 四张词典表

每张词典表都补：

- `id TEXT UNIQUE`
- `created_at TEXT NOT NULL`
- `updated_at TEXT NOT NULL`
- `deleted_at TEXT NULL`

### 3. history 表

新增：

- `sync_id TEXT UNIQUE`
- `updated_at TEXT NOT NULL`
- `deleted_at TEXT NULL`

如果保留原 `id INTEGER` 作本地索引，则同步时统一使用 `sync_id`。

### 4. callsign_qth_history 表

新增：

- `sync_id TEXT UNIQUE`
- `updated_at TEXT NOT NULL`
- `deleted_at TEXT NULL`

### 5. 本地模型

以下客户端模型或数据层都要统一输出规范化格式：

- `lib/models/log_entry.dart`
- `lib/models/dictionary_item.dart`
- `DatabaseHelper` 中 history / callsign_qth_history 的读写格式

## 服务端结构调整

### 1. 新增 history 仓储

服务端需要补齐 history 的完整模型：

- Memory adapter
- MySQL adapter
- MongoDB adapter
- Repository
- Service

支持：

- `findSince(timestamp, userId)`
- `upsert(data, userId)`
- `softDelete(id, deletedAt, userId)`

### 2. 四类数据统一同步能力

以下四类数据都要补统一接口：

- `logs`
- `dictionaries`
- `callsign_qth_history`
- `history`

每类都需要：

- 查询增量
- upsert
- 软删除
- 冲突解决

### 3. 软删除约束

服务端 API 的常规删除路径也要改为软删除，至少对同步覆盖的四类资源如此。

这意味着：

- 旧的 `delete()` 逻辑应改成设置 `deletedAt`
- 普通列表查询默认过滤 `deletedAt != null`
- 同步查询必须包含已软删除的项目，以便另一端也同步删除状态

## 双向同步协议

### 新协议

保留 `/api/v1/logs/sync/bidirectional`，但语义升级为统一双向同步入口。

请求：

```json
{
  "deviceId": "linux-desktop-01",
  "lastSyncAt": "2026-04-19T09:00:00.000Z",
  "payload": {
    "logs": [...],
    "dictionaries": [...],
    "callsignQthHistory": [...],
    "history": [...]
  }
}
```

响应：

```json
{
  "success": true,
  "serverTime": "2026-04-19T09:35:00.000Z",
  "changes": {
    "logs": [...],
    "dictionaries": [...],
    "callsignQthHistory": [...],
    "history": [...]
  },
  "mapping": {
    "client-temp-id": "server-id-or-confirmed-id"
  }
}
```

### 协议行为

1. 客户端上传本地自 `lastSyncAt` 以来的变更集
2. 服务端针对每类数据分别合并
3. 服务端返回自 `lastSyncAt` 以来服务器侧变更集
4. 客户端应用返回的服务端变更
5. 客户端以 `serverTime` 更新本地 `lastSyncAt`

## 冲突解决

### 基本规则

采用“更新时间较新者优先”：

1. 双方都存在，比较 `updatedAt`
2. 一方软删除，则比较 `deletedAt` 与另一方 `updatedAt`
3. 时间更晚的一方获胜

### 删除冲突示例

#### 情况 A

- 本地：`updatedAt=10:00`
- 服务端：`deletedAt=10:05`

结果：服务端删除胜出，客户端也标记删除。

#### 情况 B

- 本地：`deletedAt=10:00`
- 服务端：`updatedAt=10:08`

结果：服务端更新胜出，客户端撤销本地删除并应用新版本。

### 时间来源

为了避免完全依赖客户端本地时间，服务端在响应中始终返回 `serverTime`。客户端下次同步使用服务端确认过的时间戳作为 `lastSyncAt`。

## 客户端自动同步策略

### 统一同步入口

客户端 `SyncProvider` 应只保留一个底层同步入口，例如：

- `runBidirectionalSync()`

`manual` / `realtime` / `interval` 都复用它。

### 触发时机

#### 1. 建立连接后自动同步一次

在以下情况，若模式是 `realtime` 或 `interval`，自动同步一次：

- 登录成功后
- 开启同步后
- 从 `manual` 切到 `realtime` 或 `interval` 后
- 启动恢复已保存同步配置后

#### 2. 实时同步

以下操作触发一次双向同步：

- 新增日志
- 修改日志
- 软删除日志
- 新增/修改/删除词典
- 新增 history 归档
- 新增 callsign-qth 历史

#### 3. 间隔同步

按设定间隔调用统一双向同步入口。

#### 4. 手动同步

按钮仍保留，但调用同一个双向同步入口。

## 非目标

本次设计不包括：

- 多设备同时编辑同一条记录时的人工冲突确认 UI
- 共享数据与个人数据混合同步的权限扩展
- 增量字段级 merge（只做整条记录级别的最后写入获胜）

## 风险与边界

### 1. 本地数据迁移

当前客户端 SQLite 结构缺少统一同步字段，必须做迁移。迁移失败会导致旧数据无法参与双向同步。

### 2. history 大对象

`history.logsData` 可能体积较大。初版可以先保留字符串整体同步，但需要限制体积并评估后续拆分。

### 3. 旧数据兼容

现有没有 `id / updatedAt / deletedAt` 的记录，迁移时要补默认值并分配 UUID。

### 4. 服务端现有接口兼容

现有 `pushSync` / `pullSync` 可先保留，但新的客户端路径应优先切到升级后的 `bidirectional`。

## 建议实现顺序

1. **先规范客户端本地格式**：补 UUID、时间戳、软删除字段
2. **补服务端 history 模型**：三种数据库适配器都支持
3. **统一四类仓储同步接口**：findSince / upsert / softDelete
4. **升级 bidirectional 协议**：四类数据统一处理
5. **客户端接入统一双向同步入口**：manual / realtime / interval 全部走它
6. **最后收尾兼容与迁移验证**

## 结论

这次同步不是“再加几个触发点”就够了，核心是先把**客户端数据格式规范化**，再让**服务端四类数据都具备统一同步语义**。在这个基础上，realtime、interval、manual 才能真正共享同一套双向同步实现。
